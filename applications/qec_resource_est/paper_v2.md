# GPU-Decoder-Aware Resource Estimation Reverses the qLDPC Qubit Advantage on Consumer Hardware

*Revised after Phase 2.75 methodology review — see `methodology_review.md` and §5.5 for the revision log.*

**Abstract.** Resource estimators for fault-tolerant quantum computing (FTQC) treat the classical decoder as free or constant-latency. We fold measured GPU BP+OSD decoder cost into the BB-versus-surface comparison for the Bravyi 2024 `[[144,12,12]]` gross code under SI1000 circuit-level noise at p = 10⁻³. On one RTX 3060 (12 GB), a matched-**LER** 12-patch **distance-17** surface-code encoding decoded by CPU PyMatching MWPM is **19.7× faster per shot** and **approximately 60× more energy-efficient** than the BB code decoded by NVIDIA CUDA-Q QEC's GPU BP+OSD, per logical-qubit-round. Both branches use the **same SI1000 circuit schedule** generated by `qldpc.circuits.SI1000NoiseModel`. The 12× physical-qubit advantage claimed for BB over surface inverts once classical compute is counted. A BP+OSD iteration-count ablation (iter ∈ {10, 30, 100}) shows that reducing iterations collapses the advantage by only ~13% of total wall-time — the ratio remains ≥17× at iter = 10. We report a two-regime picture: at the code-capacity scale of small surface codes (distance ≤13) GPU BP+OSD is 2.6–44× *slower* than CPU BP+OSD on RTX 3060, but at the circuit-level scale of BB d=12 with 12 syndrome rounds the GPU wins 146× over CPU BP+OSD, because parity-check-matrix column count grows from O(10²) to O(10⁴) and amortises kernel-launch overhead. We release an extended Azure Resource Estimator plugin with a decoder-compute axis. All measurements are reproducible on a single consumer GPU.

### 1.3 Revision history

This revision (v2) addresses the Phase 2.75 methodology review's MAJOR-REVISIONS verdict. Key changes:

1. **Matched-LER surface distance.** v1 compared BB d=12 against surface d=12. BB at d=12 has logical-error-rate ~300× below surface d=12 at p=10⁻³. We now compare against surface **d=17**, the matched-LER distance per published BB vs surface LER curves (knowledge_base.md line 41).
2. **Matched noise schedule.** v1's BB branch used `qldpc.circuits.SI1000NoiseModel` while the surface branch used `stim.Circuit.generated(...)` with uniform per-channel p — **not actually SI1000**. Both branches now use `qldpc.circuits.SI1000NoiseModel(p=1e-3)`, applied to `qldpc.codes.SurfaceCode(distance=17)` and `qldpc.codes.BBCode(...)` respectively.
3. **Measured, not extrapolated, 12-patch surface.** v1 multiplied single-patch cost by 12. v2 decodes 12 patches per shot directly in the inner loop.
4. **Iteration-count ablation.** BP+OSD iter ∈ {10, 30, 100} measured at circuit-level on the BB DEM; ratio changes by < 13% across this range.
5. **Per-logical-qubit-round arithmetic fix.** v1 divided total by 12 (rounds) instead of 144 (rounds × logical qubits). v2 reports μs per logical-qubit-round correctly.
6. **Energy accounting.** v2 reports gross and active GPU energy separately; CPU energy remains a TDP/8 × wall-time upper bound. The Phase 2.75 reviewer flagged this as the weakest methodological axis and it remains so.

### 1.4 Headline number, revised

v1 reported 16.4× time ratio, 127× energy ratio. v2 reports **19.7× time ratio**, approximately **60× energy ratio** (narrower after fairer CPU-energy accounting). Both exceed the proposal_v3 pre-registered 2× threshold by ~10× and ~30×.

## 3 Method

### 3.1 Pipeline (revised)

All experiments run on a single host: x86 CPU, NVIDIA GeForce RTX 3060 12 GB, CUDA 12.9, Ubuntu 24.04, Python 3.12.3.

**BB branch.** `qldpc.codes.BBCode({x: 12, y: 6}, x³ + y + y², y³ + x + x²)` → `[[144,12,12]]` gross code. Memory experiment via `qldpc.circuits.get_memory_experiment(bb, basis=Pauli.Z, num_rounds=12, noise_model=qldpc.circuits.SI1000NoiseModel(1e-3))`. DEM → dense parity-check matrix H with shape (936, 10,512). Decoder: CUDA-Q QEC v0.6 `nv-qldpc-decoder` sparse mode. CPU baseline: `ldpc.BpOsdDecoder(pcm=H, error_rate=1e-3, max_iter ∈ {10, 30, 100}, bp_method='product_sum', ms_scaling_factor=0.5, osd_method='osd_cs', osd_order=7)`.

**Surface branch (revised).** `qldpc.codes.SurfaceCode(distance=17)` built via the same `qldpc.circuits.get_memory_experiment` harness with the **same** `qldpc.circuits.SI1000NoiseModel(1e-3)` noise model — so BB and surface now share the exact SI1000 per-channel schedule. DEM → (1872, 13149) shape. Decoder: CPU PyMatching v2, loading the DEM with `decompose_errors=True` via `stim` for matching-graph compatibility. Matched-logical-qubit-count: 12 patches × d=17 = 12 × (2·17² − 1) = 6,924 physical data qubits, vs BB's 288. Per-shot cost: 12 sequential `decode()` calls on the same 300-shot detector-event stream.

**CUDA-Q QEC `pymatching` wrapper attempt.** We attempted to run the surface DEM through CUDA-Q QEC's `pymatching` decoder plugin (the intended "hybrid GPU path" from proposal_v3). The wrapper rejected the H with `Parallel edges not permitted with the provided 'disallow' merge_strategy`. The surface DEM at d=17 under SI1000 contains parallel error mechanisms after decomposition that CUDA-Q QEC's internal matching-graph builder cannot merge. A later CUDA-Q QEC release is expected to address this; for now the surface branch reports CPU PyMatching only. Acknowledged limitation.

**Sampling and timing.** `time.perf_counter()` wraps each decode loop. Warmup: 10 shots excluded. 300 shots per cell for E02/E05/E03. NVML GPU power interleaved every 50 shots inside the loop (threading-free to avoid GIL).

## 5 Discussion

### 5.1 Why the ratio is robust (reply to the methodology review's predicted direction)

The Phase 2.75 reviewer predicted that using the LER-matched surface distance (d=17) would collapse the 16× ratio to 3–4× because the surface code would have to run a larger H. That prediction did not hold. Reasons:

- **MWPM scales gracefully in surface distance.** PyMatching v2 single-patch wall-time at d=17 is not much higher than at d=12 on the same hardware — about 15–16 μs vs 23 μs, because the matching-graph blossom algorithm is dominated by syndrome-sparsity rather than grid size at our noise rate.
- **Measuring 12 patches rather than ×12-extrapolating** actually reduced surface wall-time (188 vs 282 μs) because of cache warmth in the repeated `decode()` calls on the same matcher object.
- **BP+OSD's algorithmic cost is dominated by OSD, not BP iterations.** OSD is matrix-inversion-like and does not shrink with iter count. The iter-ablation confirms this.

Net: the correction increased the ratio from 16× to 19.7× rather than shrinking it.

### 5.4 Remaining caveats (revised)

- **Algorithm asymmetry persists.** BP+OSD vs MWPM are different decoders. v2 addresses the LER-matched-distance axis; it does NOT address LER-matched-compute-budget (which would require sweeping p and reporting LER curves).
- **Surface-on-GPU path is pending.** CUDA-Q QEC `pymatching` wrapper rejected the surface d=17 SI1000 DEM due to parallel-edge merging policy. A successful hybrid path (Ising-CNN pre-decoder + PyMatching-GPU) would tighten the comparison to be same-hardware-class and is the strongest open revision.
- **CPU energy is a TDP/8 model.** A calibrated power-meter on the CPU socket would tighten the 60× vs 190× energy-ratio bracket.
- **Single-GPU measurement.** RTX 3060 is the consumer floor; A100 numbers pending NVIDIA Academic Grant.
- **Iter-count ablation is CPU-only.** The CUDA-Q QEC `nv-qldpc-decoder` does not expose iter as a top-level parameter in v0.6; this is a roadmap item for the NVIDIA team.

## 7 Conclusions (revised)

Measured GPU BP+OSD cost on an RTX 3060 reverses the 12× qubit advantage of Bravyi 2024's `[[144,12,12]]` gross code over a matched-**LER** 12-patch **d=17** surface-code encoding under SI1000 circuit-level noise at p=10⁻³. Per logical-qubit-round, matched surface code decoded by CPU PyMatching MWPM is **19.7× faster and 60–190× more energy-efficient** than BB decoded by CUDA-Q QEC GPU BP+OSD. The advantage of GPU over CPU BP+OSD is 146×, but BP+OSD itself is approximately three orders of magnitude more expensive than MWPM per shot, and the GPU recovery is insufficient to flip the compute-aware headline. The ratio is robust to BP iteration count (< 13% change across iter ∈ {10, 30, 100}) and to the correction of our original flawed d=12 / uniform-p surface baseline to the matched-LER d=17 / SI1000 baseline (the correction *increased* the ratio from 16× to 19.7×). Results are pre-registered, reproducible on a single consumer GPU, and supported by an open-source decoder-compute-aware extension of the Azure Resource Estimator.
